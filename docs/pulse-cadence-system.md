# Pulse Cadence System — Руководство для Product Owner

## Что это такое

Система автоматически пишет каждому сотруднику раз в 3 дня — одно короткое сообщение в Slack с встроенным вопросом. Вопросы берутся из персонального бэклога сотрудника и покрывают четыре темы: автономия, принадлежность, рост, цель.

Цель — собрать данные по всем четырём темам за квартал, не перегружая сотрудника.

---

## Как выглядит бэклог сотрудника

У каждого сотрудника на квартал — **12 вопросов** в фиксированном порядке:

```
Автономия (3 вопроса) → Принадлежность (3) → Рост (3) → Цель (3)
```

Система идёт по ним по одному. Следующий вопрос — только после того, как предыдущий закрыт или проигнорирован.

Плюс **3 engagement-вопроса**, которые активируются в последние 14 дней квартала.

---

## Жизненный цикл вопроса

```
Ожидает → Отправлен → Закрыт
              ↓
         (48ч без ответа)
              ↓
         Ожидает (уходит в конец очереди)
```

**Закрытым** вопрос считается в двух случаях:
- Сотрудник **ответил** на вопрос в разговоре с AI (система сама это фиксирует)
- Сотрудник **сам затронул** тему в любом разговоре — без проактивного вопроса (cross-pollination)

**Проигнорированным** — если 48 часов прошло и ни одного сообщения от сотрудника не было. Вопрос уходит в конец очереди и к нему вернутся позже.

---

## Конец квартала

Последние **14 дней квартала** — система переключается на engagement-вопросы (3 штуки). Обычные 12 вопросов ставятся на паузу. Если все engagement-вопросы уже закрыты — система возвращается к обычным pending вопросам.

---

## Как тестировать

### Шаг 1 — Запустить симуляцию

Есть специальный endpoint для тестирования без ожидания реальных 3 дней:

```
POST /dev/simulate-proactive-cycle

{
  "userId": "<id сотрудника>",
  "tenantId": "<id компании>",
  "steps": 5
}
```

Он прогоняет 5 шагов бэклога подряд: для каждого шага берёт следующий вопрос, помечает как отправленный, имитирует что сотрудник проигнорировал, переходит к следующему.

**Что проверить в ответе:**
- Порядок тем: сначала идут вопросы по автономии, потом принадлежность, потом рост, потом цель
- Все 5 шагов должны быть разными вопросами

### Шаг 2 — Посмотреть в дашборде

Открой `/pulse` в дашборде. Для каждого сотрудника должен появиться прогресс-блок:

```
████░░░░  3/12 закрыто  · в очереди: 7  · проигнорировано: 2
Следующий: "Насколько ты чувствуешь связь с командой?" [принадлежность]
```

После симуляции — счётчики должны обновиться.

### Шаг 3 — Проверить реальный check-in

Если нужно проверить что реальное сообщение уходит в Slack:

1. Убедись что у тестового сотрудника есть активное окно (survey window) и подключённый Slack
2. Запусти симуляцию на 1 шаг с `"steps": 1` — это не отправляет реальное сообщение, только двигает бэклог
3. Для реальной отправки — нужно добавить job в очередь через Bull Board (попроси разработчика)

---

## Что проверить для приёмки

| Проверка | Где смотреть | Ожидаемое |
|----------|-------------|-----------|
| Порядок тем | Ответ `/dev/simulate-proactive-cycle` | autonomy → belonging → growth → purpose |
| Прогресс в дашборде | `/pulse` | Счётчики растут после симуляции |
| Игнор уходит в конец | Запустить 13+ шагов | Первый вопрос снова появляется позже |
| Конец квартала | Поменять дату / настройку tenancy | Появляются engagement-вопросы |
| Разные вопросы у разных сотрудников | Симуляция для 2 пользователей | Бэклог у каждого свой |

---

## Настройки per-компания

Если нужно изменить окно игнора или срок engagement для конкретной компании — скажи разработчику, он обновит поле `proactive_messaging_policy` для тенанта:

| Параметр | По умолчанию | Описание |
|----------|-------------|----------|
| `ignoreWindowHours` | 48 часов | Сколько ждать ответа до считать ignored |
| `engagementUnlockDays` | 14 дней | За сколько дней до конца квартала включать engagement-вопросы |

---

## Что делать если что-то не так

**Вопросы идут не в том порядке** — скорее всего бэклог уже инициализирован с другим порядком. Попроси разработчика сбросить бэклог для конкретного пользователя.

**В дашборде не видно прогресса** — убедись что у сотрудника есть активный survey window (статус `active`). Бэклог показывается только для активного квартала.

**Симуляция возвращает пустой массив** — у сотрудника нет активного window или нет вопросов в бэклоге. Нужно сначала запустить инициализацию квартала.
